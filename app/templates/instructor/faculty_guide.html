Faculty Guide

Everything you need to set up and run the Class Attendance Tracker in your UO SOJC courses.

About This Tool

This is an open-source attendance tracker built specifically for UO School of Journalism and Communication faculty. It helps you record who showed up and collect a short learning reflection from each student as proof of attendance — all in one quick workflow.

The tool is privacy-first: it stores only student names, not UO IDs or email addresses. Each faculty member runs their own isolated database instance, so your course data is never shared with others. Built and maintained by Andrew DeVigal.

Features

  • Import your roster from DuckWeb (PDF or Excel) — or paste names manually
  • Permanent per-course QR code — print it once and reuse it every session
  • Timed attendance window — you control when it opens and closes
  • Students submit a short learning reflection as proof of attendance
  • Anti-gaming measures: time window enforcement, duplicate detection, IP flagging, and reflection quality check
  • Instructor dashboard with a color-coded attendance grid across all sessions
  • Click any cell to manually override status: P Present / A Absent / P* Flagged
  • Add instructor notes to any individual submission
  • Configurable attendance policy — set free absences and grade deduction percentage
  • End-of-term Excel export with attendance grid, summary statistics, and student reflections

How to Use It (Step by Step)

  1. 1
    Go to the live tool. Open YOUR_RENDER_URL in your browser. Bookmark this — it's your home base for all your courses.
  2. 2
    Set up a new course. Click "+ New Course" in the nav. Upload your DuckWeb class list (PDF or Excel) — or switch to manual entry and paste names directly.
  3. 3
    Configure your attendance policy. Set the number of free absences allowed, the grade deduction per excess absence, and the default session window length.
  4. 4
    Find your permanent QR code. You'll land on your Course Dashboard after setup. Your QR code is listed there — download and print it. This code never changes, so you only need to print it once per course.
  5. 5
    Before each class: open a new session. Click "Start New Session" from your Course Dashboard. Do this at least 5 minutes before class starts. Render's free tier may need a moment to wake up if the app has been idle.
  6. 6
    Display the QR code. Project it or hold up the printed copy. Students scan it and submit their reflection. The attendance window is timed — submissions outside the window are rejected.
  7. 7
    Monitor live check-ins. The Session Detail page updates in real time. Review any flagged submissions and accept or reject them. Flags may appear for late submissions, suspected duplicates, or low-quality reflections.
  8. 8
    Close the session. Click "Close Session" when class is over. The window closes automatically too, but closing manually is good practice.
  9. 9
    At end of term: export your report. From your Course Dashboard, click "Export Excel". The file includes the full attendance grid, per-student summary, and all submitted reflections.

Set Up Your Own Instance

Each faculty member runs their own private deployment — your students' data never touches anyone else's instance. Setup takes about 15 minutes.

  1. Go to github.com/drewvigal/class-attendance and click Fork to create your own copy of the repo.
  2. Sign up at render.com and connect your GitHub account when prompted.
  3. Click New → Web Service and select your forked repository. Render will detect the configuration automatically.
  4. Add two Environment Variables in Render's dashboard:
    • ANTHROPIC_API_KEY — get this from console.anthropic.com (used for reflection quality checks)
    • DATABASE_URL — create a free PostgreSQL database at neon.tech and paste the connection string here
  5. Click Deploy. Your private instance will be live at a unique Render URL within a few minutes.
  6. Copy that Render URL and bookmark it — that's your YOUR_RENDER_URL from Step 1 above.

Privacy Notes

Names only Only student names are stored — no UO ID numbers, no email addresses.
Hashed IP addresses IP addresses used for duplicate detection are hashed one-way and never stored in plain text.
Your data, your database Data lives in your own Neon PostgreSQL instance — it is never shared with other faculty deployments.
No tracking or analytics No third-party analytics, cookies, or tracking scripts are used in this tool.
